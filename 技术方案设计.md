# Stack Overflow RAG QA 系统技术方案设计

## 1. 文档说明

### 1.1 文档目标
本文档基于《项目计划书》进一步细化技术实现方案，面向后续项目创建、架构设计、任务拆分与编码落地，重点解决以下问题：

- 系统应如何分层与拆模块。
- 跨语言 RAG 链路应如何实现。
- 数据如何采集、清洗、入库和更新。
- 在线问答接口应如何组织。
- 如何控制回答质量、性能、成本与可维护性。

### 1.2 适用范围
本文档适用于 Stack Overflow RAG QA 系统首期版本，默认采用：

- 英文 Stack Overflow 作为首期知识源。
- 中文提问 / 中文回答作为默认交互模式。
- Python 技术栈作为默认实现方向。

### 1.3 设计原则

- 先闭环后优化：优先跑通最小可用跨语言 RAG 链路。
- 检索优先于生成：回答质量首先取决于召回质量。
- 保留可追溯性：所有回答必须能追溯到原始帖子。
- 尽量少做不可逆转换：知识库保留英文原文，不做离线全量翻译。
- 模块解耦：便于后续替换 Embedding、向量库、LLM 与重排序策略。

## 2. 总体技术架构

### 2.1 架构概览
系统分为离线构建链路和在线问答链路两部分。

```text
离线链路：
Stack Overflow 数据采集
  -> 数据清洗与标准化
  -> 文档切分与知识单元生成
  -> Embedding 批量向量化
  -> ChromaDB / FAISS 索引构建
  -> MySQL 元数据入库

在线链路：
用户提问
  -> 问题预处理
  -> 语言识别
  -> 中文查询改写为英文检索表达
  -> Embedding 生成
  -> 向量召回
  -> 重排序
  -> Context 构建
  -> LLM 生成中文答案
  -> 引用溯源与结构化返回
```

### 2.2 分层设计

#### 2.2.1 接入层
负责接收用户请求和返回结果。

建议包含：

- Web UI 或内部管理界面。
- FastAPI 问答接口。
- 健康检查接口。
- 管理接口。

#### 2.2.2 应用编排层
负责串联整个问答流程。

建议包含：

- 问答主流程 Orchestrator。
- 检索编排器。
- Prompt 构建器。
- 回答结构化处理器。

#### 2.2.3 检索层
负责向量化、召回与排序。

建议包含：

- Query Embedding 服务。
- ChromaDB 检索服务。
- FAISS 原型检索服务。
- Rerank 服务。
- 检索缓存。

#### 2.2.4 数据层
负责原始数据、结构化文档、向量索引和元数据管理。

建议包含：

- 本地原始数据存储。
- 知识单元存储。
- ChromaDB 向量库。
- MySQL 数据库。
- Redis 缓存。

#### 2.2.5 模型层
负责 Embedding 和 LLM 调用。

建议包含：

- Embedding Provider 适配层。
- LLM Provider 适配层。
- 模型路由与降级逻辑。

#### 2.2.6 运行与评估层
负责日志、评测和运行保障。

建议包含：

- Python logging 日志记录。
- `print` 调试输出。
- 离线评测。
- 人工抽检。

## 3. 技术选型建议

### 3.1 默认技术栈

- 语言：Python 3.11+
- Web 框架：FastAPI
- 数据校验：Pydantic
- 异步任务：Celery
- 消息中间件：Redis
- 数据库：MySQL
- 向量库：ChromaDB
- 本地原型索引：FAISS
- 文件存储：本地文件系统
- 日志 / 监控：Python logging + print 调试

### 3.2 模型选型建议

#### 3.2.1 Embedding
首期建议采用以下思路：

- 知识库文档使用适合英文技术文本的 Embedding 模型。
- 中文问题先做查询改写，再进行英文向量检索。
- 第二阶段引入跨语种 Embedding 作为补充召回通道。

首期建议实现为双模式：

- 模式 A：`rewrite_to_en -> embedding -> retrieve`
- 模式 B：`zh_query -> multilingual_embedding -> retrieve`

上线初期默认启用模式 A，模式 B 作为灰度增强项。

#### 3.2.2 LLM
LLM 需满足以下要求：

- 能根据英文上下文生成高质量中文答案。
- 能严格遵循“只依据上下文回答”的约束。
- 支持结构化输出。
- 支持稳定的低温度生成。

#### 3.2.3 Rerank
首期可采用两类策略：

- 轻量规则重排序。
- 可选 Cross-Encoder 语义重排序。

推荐演进顺序：

- PoC：规则重排序
- 内测：规则重排序 + 可选 Cross-Encoder 灰度
- 生产：按成本与效果决定是否默认启用 Cross-Encoder

## 5. 数据模型设计

### 5.1 原始帖子模型
建议保留原始问题和答案的核心字段：

```json
{
  "question_id": 79320365,
  "title": "Unable to deserialize the data into TValue",
  "question_body": "...",
  "tags": ["c#", ".net", "redis"],
  "creation_date": 1735661169,
  "score": 0,
  "view_count": 44,
  "answer_count": 0,
  "accepted_answer_id": null,
  "link": "https://stackoverflow.com/questions/79320365/...",
  "answers": []
}
```

### 5.2 知识单元模型
向量检索不建议直接以“整条原始问答”粗粒度入库，建议生成知识单元：

```json
{
  "doc_id": "so_79320365_a0_chunk_0",
  "question_id": 79320365,
  "answer_id": null,
  "doc_type": "question_answer_unit",
  "title": "Unable to deserialize the data into TValue",
  "content": "title + question + accepted answer/selected answer",
  "tags": ["c#", ".net", "redis"],
  "score": 0,
  "is_accepted": false,
  "source_url": "https://stackoverflow.com/questions/79320365/...",
  "language": "en",
  "created_at": "2025-01-01T00:00:00Z",
  "updated_at": "2025-01-01T00:00:00Z"
}
```

### 5.3 查询日志模型
建议记录问答请求全链路日志：

```json
{
  "query_id": "uuid",
  "user_query": "Redis 批量写入后反序列化报错怎么办",
  "detected_language": "zh",
  "rewritten_query": "redis bulk write deserialization error TValue",
  "retrieved_doc_ids": ["doc1", "doc2"],
  "final_answer": "...",
  "latency_ms": 2480,
  "llm_model": "model-x",
  "embedding_model": "embed-y",
  "created_at": "2026-06-05T15:30:00Z"
}
```

### 5.4 术语映射模型
建议维护术语映射表，支持查询改写和回答术语统一：

```json
{
  "term": "依赖注入",
  "english_term": "dependency injection",
  "category": "framework_concept",
  "enabled": true

```

## 6. 离线数据构建方案

### 6.1 数据导入流程

```text
下载 Stack Overflow 数据
  -> 解析 JSONL / API 结果
  -> 清洗 HTML
  -> 保留问题与答案关系
  -> 生成知识单元
  -> 批量 Embedding
  -> 写入 ChromaDB
  -> 写入 MySQL
```

### 6.2 数据清洗规则

- 去除 HTML 标签和无意义空白。
- 保留代码块、错误日志块和标题结构。
- 过滤明显空问题、垃圾内容和超低价值内容。
- 合并问题标题与问题正文。
- 优先选用最佳答案，如无最佳答案则选高票答案。

### 6.3 文档切分规则

- 问题较短：标题 + 正文 + 最佳答案组成单条知识单元。
- 问题较长：保留问题上下文，答案按段落切分。
- 代码与说明文字尽量同片段存储。
- 每个片段控制在适合 Embedding 和 LLM 上下文拼装的长度范围内。

### 6.4 索引构建策略

- 全量初始化：用于首次建库。
- 增量更新：用于新数据追加。
- 索引版本化：支持新旧索引灰度切换。

建议保留以下能力：

- `rebuild_full_index()`
- `append_incremental_docs()`
- `switch_active_index(version)`

## 7. 在线问答链路设计

### 7.1 主流程

```text
1. 接收问题
2. 清洗文本
3. 检测语言
4. 中文问题改写为英文检索表达
5. 生成查询向量
6. 进行向量召回
7. 执行重排序
8. 构建 Context
9. 调用 LLM 生成中文答案
10. 注入引用来源
11. 返回结构化结果
```

### 7.2 问题预处理策略

建议提取以下信号：

- 编程语言。
- 框架名。
- 中间件名。
- 报错信息。
- 类名 / 方法名 / API 名。
- 版本号。

预处理输出建议：

```json
{
  "raw_query": "Redis 批量写入后反序列化报错怎么办",
  "normalized_query": "Redis 批量写入后反序列化报错 怎么办",
  "language": "zh",
  "keywords": ["Redis", "反序列化"],
  "error_terms": [],
  "versions": []
}
```

### 7.3 查询改写策略
对中文问题进行面向检索的英文改写，而不是普通机器翻译。

目标：

- 保留技术术语。
- 补齐检索关键词。
- 去掉口语化表达。
- 输出更接近英文技术社区问法的检索表达。

示例：

- 中文原问：
  - `Redis 批量写入后反序列化报错怎么办`
- 改写后：
  - `redis bulk write deserialization error TValue`

### 7.4 召回策略

#### 7.4.1 单路召回
首期最简方案：

- 仅使用英文改写后的 query 做向量召回。

#### 7.4.2 双路召回
增强方案：

- 路由 1：中文改写后的英文 query
- 路由 2：原始中文 query 的跨语种向量

合并逻辑：

- 合并去重。
- 保留最高相关度分数。
- 交由 rerank 统一排序。

### 7.5 重排序策略
首期建议结合规则信号：

- 向量相似度。
- 标题匹配度。
- 标签重合度。
- 错误关键词命中。
- 答案分数。
- 是否最佳答案。

重排序输出建议为：

```json
[
  {
    "doc_id": "doc_1",
    "score": 0.91,
    "title": "Unable to deserialize the data into TValue"
  }
]
```

### 7.6 Context 构建策略
Context 不是简单拼接全文，建议采用结构化模板：

```text
[Document 1]
Title: ...
Tags: ...
Question Summary: ...
Answer Excerpt: ...
Source: ...
```

构建原则：

- 优先保留标题与最佳答案。
- 控制总 Token 数。
- 删除重复或低价值片段。
- 保证每段都保留来源映射。

### 7.7 答案生成策略
回答生成应遵循以下规则：

- 默认输出中文。
- 关键错误信息、命令、类名、配置项保留英文原文。
- 不得编造上下文中不存在的结论。
- 上下文不足时明确说明“不足以确认”。

建议输出结构：

```json
{
  "answer": "中文回答",
  "confidence": "medium",
  "citations": [
    {
      "title": "...",
      "url": "https://stackoverflow.com/..."
    }
  ],
  "notes": [
    "关键术语保留英文原文"
  ]
}
```

## 8. API 设计建议

### 8.1 问答接口

`POST /api/v1/qa/ask`

请求示例：

```json
{
  "query": "Redis 批量写入后反序列化报错怎么办",
  "top_k": 5,
  "return_context": false
}
```

响应示例：

```json
{
  "query_id": "uuid",
  "answer": "从召回结果看，这类问题通常与 JSON 结构和反序列化目标类型不一致有关。",
  "confidence": "medium",
  "citations": [
    {
      "title": "Unable to deserialize the data into TValue",
      "url": "https://stackoverflow.com/questions/79320365/..."
    }
  ],
  "debug": {
    "language": "zh",
    "rewritten_query": "redis bulk write deserialization error TValue"
  }
}
```

### 8.2 健康检查接口

- `GET /api/v1/health`
- `GET /api/v1/ready`

### 8.3 管理接口

- `POST /api/v1/admin/rebuild-index`
- `POST /api/v1/admin/ingest`
- `GET /api/v1/admin/index/versions`
- `POST /api/v1/admin/index/switch`

## 9. Prompt 设计建议

### 9.1 系统提示词目标

- 明确回答必须基于提供的上下文。
- 明确输出语言为中文。
- 明确保留关键英文技术术语。
- 明确给出来源引用。

### 9.2 Prompt 样例结构

```text
你是一个技术问答助手。
请仅根据提供的 Stack Overflow 检索上下文回答用户问题。
如果上下文不足，请明确说明。
回答使用中文，但保留关键英文术语、报错信息、类名、方法名、配置项原文。
输出需包含：
1. 结论
2. 处理建议
3. 参考来源
```

## 10. 缓存设计

### 10.1 缓存对象

- 中文问题预处理结果。
- 中文查询改写结果。
- Query Embedding 结果。
- Top-K 检索结果。
- 最终回答结果。

### 10.2 缓存 Key 建议

- `preprocess:{hash(query)}`
- `rewrite:{hash(query)}`
- `embedding:{model}:{hash(query)}`
- `retrieve:{index_version}:{hash(query)}`
- `answer:{prompt_version}:{hash(query)}`

### 10.3 缓存策略

- 改写结果可较长时间缓存。
- 检索结果按索引版本隔离。
- 最终答案缓存需绑定 Prompt 版本和索引版本。

## 15. 结论
本技术方案设计以“英文知识库 + 中文提问/中文回答”为核心路线，兼顾首期快速落地与后续可扩展性。整体实现重点不在离线翻译整个知识库，而在构建稳定的跨语言检索桥接能力、可靠的上下文控制能力和可追溯的中文回答生成能力。

从工程实践上，建议优先完成最小可用闭环，再逐步引入双路召回、跨语种 Embedding、重排序、缓存和评测体系。这样既能尽快启动项目编码，也能为后续质量优化预留清晰演进路径。
