# Stack Overflow RAG QA 系统 — 项目分析

> 面向 Agent 开发新手的全项目分析报告

---

## 一、项目目的与目标

### 项目全称
Stack Overflow RAG QA 系统

### 核心目的
构建一个基于 **RAG（检索增强生成）** 的智能问答系统，让用户能用 **中文或英文** 自然语言提问技术问题，系统自动从 Stack Overflow 英文知识库中检索相关内容，并生成 **带引用来源的中文结构化回答**。

### 关键目标
- 降低英文技术知识的检索和阅读门槛（中文提问 → 英文知识库 → 中文回答）
- 所有回答可追溯到原始 Stack Overflow 帖子，降低 LLM 幻觉风险
- 分层解耦架构，便于替换 Embedding 模型、向量库和 LLM
- 按 PoC → 内测版 → 生产版三阶段迭代交付

## 二、架构模式与设计

项目采用 **分层架构 + 依赖注入 + Provider 模式**。

### 分层结构（7 层）

```
接入层 (API Routes)          → FastAPI 路由，接收请求/返回响应
    ↓
应用编排层 (QAService)        → 串联整个问答流程的 Orchestrator
    ↓
服务层 (Services)             → 预处理、改写、检索、重排、上下文、答案、引用
    ↓
Provider 层                   → LLM / Embedding / VectorStore 的抽象适配
    ↓
Repository 层                 → 文档加载、查询日志持久化
    ↓
数据层                        → JSONL 种子数据、ChromaDB 索引、本地 JSON 缓存/日志
    ↓
运行评估层                    → 结构化日志、评测脚本
```

### 关键设计模式

| 模式 | 用途 | 体现位置 |
|------|------|----------|
| **Provider 模式** | 抽象第三方服务，支持 Mock ↔ 真实云端一键切换 | `app/providers/` |
| **依赖注入** | `main.py` 的 `create_app()` 中组装所有服务 | `app/main.py` |
| **工厂模式** | `build_vector_store()` 根据配置决定用 in_memory 还是 ChromaDB | `providers/vector_store/factory.py` |
| **Repository 模式** | 数据访问与业务逻辑分离 | `app/repositories/` |
| **策略模式** | 双路召回策略可配置开关 | `qa_service.py` 中 `enable_dual_retrieval` |

## 三、关键组件与职责

### 3.1 服务层 (app/services/) — 8 个服务

| 服务 | 职责 | 核心逻辑 |
|------|------|----------|
| **PreprocessService** | 问题预处理 | 去空白、检测语言（正则判断中文字符）、提取关键词/错误词/版本号 |
| **RewriteService** | 查询改写（中文→英文检索表达） | 优先用 LLM 改写；降级为术语表替换（TERM_MAP 字典） |
| **RetrievalService** | 向量检索 | 单路/双路召回、结果合并去重（保留最高分+记录路由来源） |
| **RerankService** | 规则增强重排序 | 向量分 + 标签匹配(+0.05) + 标题匹配(+0.1) + 最佳答案(+0.05) + 答案分数 + 多路由命中(+0.08) |
| **ContextService** | 构建 LLM 输入上下文 | 结构化模板：Title → Tags → Answer Excerpt → Source |
| **AnswerService** | 调用 LLM 生成答案 | 委托给 LLMProvider，附带系统 Prompt 约束 |
| **CitationService** | 构建引用列表 | 从排序结果提取 Title + URL，去重，最多 3 条 |
| **CacheService** | 答案缓存 | 内存字典 + JSON 文件持久化，TTL 过期机制 |

### 3.2 Provider 层 (app/providers/) — 3 大类

**LLM Provider**:
- `LLMProvider` (抽象基类) → 定义 `generate_answer()` 和 `rewrite_query()`
- `MockLLMProvider` → Mock 实现，基于召回内容生成摘要
- `OpenAICompatibleLLMProvider` → 通过 httpx 调用 OpenAI 兼容 API

**Embedding Provider**:
- `EmbeddingProvider` (抽象基类) → 定义 `embed()` 和 `embed_batch()`
- `MockEmbeddingProvider` → Mock 实现，基于词频的伪向量
- `OpenAICompatibleEmbeddingProvider` → 调用 OpenAI 兼容 Embedding API，支持批量（每批10条）

**Vector Store**:
- `VectorStore` (抽象基类) → 定义 `build()` 和 `search_by_vector()`
- `InMemoryVectorStore` → 内存余弦相似度检索
- `ChromaVectorStore` → ChromaDB 持久化向量库，距离转分数公式：`1 / (1 + distance)`

### 3.3 数据模型 (app/models/)

- **Document**: 知识单元，包含 doc_id、标题、内容、标签、分数、是否最佳答案、来源链接等
- **QueryLog**: 查询日志，记录全链路信息（用户问题、语言、改写结果、召回文档、最终答案、延迟、模型名等）

### 3.4 路由层 (app/api/routes/)

| 路由 | 接口 | 功能 |
|------|------|------|
| `qa.py` | `POST /api/v1/qa/ask` | 核心问答接口 |
| `admin.py` | `GET /api/v1/admin/query-logs` | 查询日志列表 |
| `admin.py` | `GET /api/v1/admin/cache/stats` | 缓存统计 |
| `admin.py` | `POST /api/v1/admin/rebuild-index` | 重建索引 |
| `admin.py` | `GET /api/v1/admin/index/versions` | 索引版本列表（预留） |
| `health.py` | `GET /api/v1/health`、`/ready` | 健康检查 |

## 四、数据流：一次 QA 请求的完整生命周期

```
用户发起 POST /api/v1/qa/ask
  { "query": "Redis 批量写入后反序列化报错怎么办", "top_k": 5 }
        │
        ▼
  ① 答案缓存检查 ──命中──→ 直接返回缓存结果
        │ 未命中
        ▼
  ② PreprocessService: 文本清洗 + 语言检测
        → normalized_query = "Redis 批量写入后反序列化报错怎么办"
        → language = "zh"
        │
        ▼
  ③ RewriteService: 中文改写为英文检索表达
        → rewritten_query = "redis bulk write deserialization error"
        (通过 LLM 或术语表替换)
        │
        ▼
  ④ RetrievalService: 双路召回（因为 language=="zh" 且 enable_dual_retrieval==true）
        路由1: rewrite_to_en → 用英文改写查询向量检索
        路由2: zh_query → 用原始中文查询向量检索
        │
        ▼
  ⑤ 合并去重: merge_results()
        → 同一文档保留最高分，记录命中了哪些路由
        │
        ▼
  ⑥ RerankService: 规则增强重排序
        → 向量分 + 标签/标题/最佳答案/多路由加分
        │
        ▼
  ⑦ ContextService: 构建 LLM 上下文（Top-4 文档结构化拼装）
        → "[Document 1] Title: ... Tags: ... Answer Excerpt: ... Source: ..."
        │
        ▼
  ⑧ AnswerService: 调用 LLM 生成中文答案
        → System Prompt: "你是一个技术问答助手，仅根据上下文回答..."
        │
        ▼
  ⑨ CitationService: 提取引用来源（Title + URL）
        │
        ▼
  ⑩ 记录查询日志 + 写入答案缓存
        │
        ▼
  返回 AskResponse:
    {
      "query_id": "uuid",
      "answer": "中文回答...",
      "confidence": "medium",
      "citations": [{ "title": "...", "url": "..." }],
      "notes": ["回答基于当前检索上下文生成。"],
      "debug": { "language": "zh", "rewritten_query": "...", ... }
    }
```

## 五、技术栈

| 类别 | 技术 | 用途 |
|------|------|------|
| **语言** | Python 3.11+ | 开发语言 |
| **Web 框架** | FastAPI | HTTP API 服务 |
| **数据校验** | Pydantic + pydantic-settings | 请求/响应模型 + 配置管理 |
| **向量库** | ChromaDB (主) / In-Memory (备选) | 向量检索存储 |
| **LLM 接入** | OpenAI Compatible API (httpx) | 答案生成 + 查询改写 |
| **Embedding** | OpenAI Compatible API (httpx) | 文本向量化 |
| **日志** | Python logging | 结构化日志 |
| **缓存** | 内存 dict + JSON 文件 | 答案缓存 |
| **持久化** | 本地 JSONL 文件 | 查询日志、缓存数据 |
| **HTTP 客户端** | httpx | 调用 LLM/Embedding API |
| **配置管理** | .env 文件 + 环境变量前缀 `SO_RAG_` | 全局配置 |

## 六、开发三阶段

### 第一阶段：PoC 验证

- 项目骨架搭建，FastAPI 基础接口
- 最小可用 RAG 闭环（预处理 → 改写 → 检索 → 重排 → 上下文 → 答案 → 引用）
- 本地 JSONL 种子数据加载 + 回退文档
- Mock Provider 实现（无需真实 API 即可运行）
- 导入、重建索引、检索评估脚本

### 第二阶段：内测版增强

- **双路召回**：英文改写召回 + 中文原问召回
- **合并去重** + 增强重排序（多信号融合）
- **答案缓存**：内存 + JSON 文件持久化，TTL 过期
- **结构化查询日志**：全链路追踪 + 管理接口
- **可切换向量库**：`in_memory` / `chroma` 通过工厂模式切换
- **ChromaDB 本地持久化索引**

### 第三阶段：生产版 🚧 预留

- 索引版本化与灰度切换
- 增量更新能力
- Redis 替代内存缓存
- Cross-Encoder 真实重排序
- 完善异常处理、降级策略与运维手册

## 八、总结

这是一个**结构清晰、设计良好**的 RAG 系统原型项目，采用了标准的工程实践：

- **Provider 模式**让 Mock ↔ 真实云端切换只需改 `.env` 配置
- **依赖注入**在 `create_app()` 中集中组装，便于测试和替换
- **双路召回 + 多信号重排序**是跨语言 RAG 的核心亮点
- **全链路日志**让每次查询的每一步都可追溯
- 当前代码处于**第二阶段内测完成**状态，核心链路已跑通，正在等待接入真实云端模型后进行质量验证
